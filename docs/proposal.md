# 企画書: s3Ollama バッチ推論プラットフォーム

## 背景

社内で管理される大量のドキュメントやテキスト資産を活用し、ジェネレーティブ AI による要約・分析・コード生成を行いたいというニーズが増加しています。しかし、オンデマンドに GPU リソースを確保して推論ジョブを安全に実行できる仕組みが不足しており、都度の環境構築が運用負荷となっています。

## 目的

- S3 バケットに入力を配置し API をコールするだけで推論が走る、セルフサービス型の推論基盤を構築する。
- 推論結果を自動で S3 に保存し、完了通知を SNS 経由で配信することで利用者のフィードバックループを短縮する。
- Spot インスタンス活用により GPU コストを最適化する。

## スコープ

- Terraform による AWS リソースの自動構築
- API Gateway + Lambda による Batch ジョブのトリガー
- AWS Batch (スポット) を利用した Ollama 推論ジョブの実行
- 入出力 S3 バケットおよび SNS 通知の整備
- 利用者/運用者向けドキュメントの整備

## 成果物

- Terraform コード一式
- Lambda (API) 実装
- Batch 実行ランナー実装
- 利用者向け/運用者向けドキュメント
- インフラ構成図 (Mermaid)

## ターゲットユーザー

- 部門横断でバッチ推論を実行したい開発者・データサイエンティスト
- 推論結果をレポートとして受け取りたいビジネスユーザー
- プラットフォームを運用する SRE/インフラチーム

## 要件

1. **入力管理**: ユーザーは指定の S3 バケットに `contexts/` と `prompt.txt` を含むディレクトリをアップロードする。
2. **API 連携**: REST API で入力プレフィックスを指定するとバッチ処理が開始される。
3. **推論実行**: AWS Batch で Spot インスタンスを利用し Ollama モデル (例: llama2) を実行する。
4. **出力管理**: 結果を別 S3 バケットに保存し、入力と同じプレフィックスに配置する。
5. **通知**: SNS トピックに処理完了 (成功/失敗) を通知する。
6. **運用性**: Terraform による環境複製、CloudWatch Logs でのモニタリング、パラメータ調整の柔軟性を確保する。

## アーキテクチャ概要

- ユーザーが入力ファイルを S3 (input) にアップロード
- API Gateway 経由で Lambda が AWS Batch ジョブを起動
- Batch ジョブ (Ollama コンテナ) が S3 input を取得し、推論結果を S3 output に保存
- SNS で完了通知を配信

## スケジュール案 (イメージ)

| フェーズ | 期間 | 内容 |
| -------- | ---- | ---- |
| 設計 | 1 週間 | 要件確定、アーキテクチャ設計、Terraform 設計 |
| 実装 | 2 週間 | Terraform 作成、Lambda・Batch ランナー実装、CI/CD 整備 |
| 検証 | 1 週間 | ステージング環境での動作確認、負荷試験、ドキュメント整備 |
| 展開 | 1 週間 | 本番デプロイ、運用チーム引き継ぎ |

## リスクと対応策

| リスク | 影響 | 対応策 |
| ------ | ---- | ------ |
| Spot 中断によるジョブ失敗 | 中程度 | 再実行手順の整備、重要ジョブにはオンデマンドを選択できるようパラメータ化 |
| モデル実行時間のばらつき | 高 | `timeout_seconds` の調整、ジョブ分割のガイドライン整備 |
| 入力データの肥大化 | 中 | S3 プレフィックスあたりの容量制限、前処理の自動化を将来検討 |
| セキュリティ確保 | 高 | IAM ロールの最小権限設計、API 認証の追加対応をロードマップ化 |

## 今後の拡張余地

- Step Functions を用いたエラーハンドリングとリトライ制御
- 出力データのメタデータ管理 (DynamoDB など)
- 監査ログ収集とダッシュボード化
- モデルバージョンの自動更新 (CI/CD 連携)

本企画により、社内で統一されたジェネレーティブ AI バッチ推論プラットフォームを短期間に立ち上げ、コスト効率と運用性を両立させることが可能になります。
